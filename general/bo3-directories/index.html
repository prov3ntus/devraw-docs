BO3 Directories & File Overrides

This section focuses on Black Ops III's directory system, and how dragging & dropping works.

It also covers which folders the compiler prioritises when building your map (map-specific file overrides).

Beginner

About directories​

When you installed Mod Tools, you may have noticed more folders in your BO3 root. Some of them are: map_source, model_export, share, sound_assets and texture_assets. These are some of the main folders that store assets for the game. You'll be using them a lot, regardless of whether you're a mapper, scripter, etc. and you'll naturally get more familiar with them over time.

Sometimes, specific files must be inside the correct folder for the game to read them.

Visualisation of the directory tree
Call of Duty Black Ops III
    └── sound_assets
        └── pv
            └── motd_announcer_finn
                └── max_ammo.wav

For example, consider the directory above. These are custom sounds (you won't have them in your mod tools directory), and I want to put them in the game. Sound files must be within sound_assets as the linker is searching <path to root>\Call of Duty Black Ops III\sound_assets\ PLUS whatever the file path for the sound is. In the case above, the full path would be:

"<path to root>\Call of Duty Black Ops III\sound_assets\" PLUS "pv\motd_announcer_finn\max_ammo.wav"

This is just because the devs wanted all sounds to be located inside the sound_assets folder. However this isn't the case for all files used by the game, it varies based on which file type we're talking about. For example, .xmodel_bin files can only be put in <BO3 ROOT>\model_export\**, but textures (PNGs, TIFFs, etc.) can go anywhere in <BO3 ROOT>\**.

But I still recommend to keep files in their respective folders. Files can get very messy very quickly, and could cause you grief later on if there's a lack of organisation, which is why in game development you'll never find a company that doesn't have a logical data structure for their game files.

Note for GDT files

GDT files are a unique case where it only looks inside the folders specified in the <BO3 root>\bin\converter_gdt_dirs_0.txt file.

As an example, if you've installed any of Kingslayer Kyle's packs, your file will have the _custom folder at the top of this file like I do. He stores GDTs in the <BO3 ROOT>\_custom\** folder, and he puts the folder name in this file so the game will read the GDTs in <BO3 ROOT>\_custom\**.

warning

You will never really need to use this. I'm explaining this here for your understanding. It's not recommended to add folders to the root directory. It's not even necessary. Stick to organising your custom GDTs properly, keeping them inside the default folders, and next to their respective assets/files. So, if you have a GDT you're putting xmodels in, it's good practice to put the GDT in the same folder in model_export as the .xmodel_bin file.

I recommend you make a custom folder of your username (or an abbreviation of it) in the main folders when you add assets to them, so you know which stuff is yours. Putting an underscore at the start will also help you find it easier, as File Explorer shows it at the top when you're sorting by name. For example: model_export\_pv\**, texture_assets\_pv_mtl\**

Dragging & Dropping (Installing Asset Packs)​

Dragging & dropping is an easy way to install someone else's assets they've released. It's a standard way for sharing assets within the Mod Tools community. The person packages up all the files by mirroring the directory they were in (putting them all into the same folders they were originally in). So when you install it, you just drag the folders from their pack into your BO3 root, and the files will be in the same place as the person who shared the files.

The file paths will all be the same, so if I'm releasing that Mob of the Dead Announcer pack, I would put it in the same directory tree. Then when the game compiles your map, it will find the files in the same location (from the BO3 root folder)

Always check for instructions

People usually leave instruction on how to install their pack, in case there are a few more steps than a simple drag n' drop.

Make sure you're checking for the instructions and reading through them thoroughly so you don't miss anything. No one likes helping a guy in help chat who was too lazy to read instructions properly. If you genuinely need some help tho that's perfectly okay.

File overrides​

Just like sound_assets is the base folder for sound files, there is another 'base' folder used for common shared files used by all maps, located in <BO3 root>\share\**. There are three folders in here: assetconvert\, gamedata\, zone_source\ & raw\.

  • assetconvert\ is where the assets that are "converted" (compiled) are cached, for example when you drag a model down in Radiant, it has to convert the model into a mesh and compress its textures. When it does those conversions, it caches the converted files so you won't have to do that conversion process again if you decide to use the model elsewhere.

  • gamedata\ is mainly used for vox stuff, which you don't need to know right now.

  • zone_source\ is where the game looks for .zpkg files (see Zone Files if you don't know what .zpkg files do).

  • And, last but not least, the raw\ subfolder in the share\ folder is where most of the global folders are stored. We're gonna be focusing on this folder for now.

Now there are a ton of folders here, but don't let that intimidate you. For now, let's just focus on the scripts folder.

This is where all the global scripts are stored. What do I mean by global? Well, these scripts can be called and used in any map (assuming you've zoned them, of course). This also applies to every other file in the share\raw\** directory.

But what if you wanted to change a script, or another file for a specific map? For example, you wanted to change the perk machines to different models in only one map? Because if you change the script in share\raw\**, it'll reflect the changes whenever you build any map using that script.

Well, before the linker checks if the file is in share\raw\**, it'll actually check if it's in your map folder first (e.g. <BO3 root>\usermaps\zm_ambient_test\). The root of your map folder will mirror the share\raw\** folder.

Let's keep going with that perk machine problem. I only want to change the model for, let's say, quick revive in my zm_ambient_test map. The <BO3 ROOT>\usermaps\zm_ambient_test\ folder acts as a map-specific override above anything in share\raw\**. In this case I want to change the perk machine model, so I would copy the quick revive .gsh file from <BO3 ROOT>\share\raw\scripts\zm\*, and paste it here: <BO3 ROOT>\usermaps\zm_ambient_test\scripts\zm\*.

Now, I can change what I like about the script I just copied, and it will only apply to the respective level (zm_ambient_test in this case).
