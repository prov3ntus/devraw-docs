Glossary

Here, you'll find a list of definitions for terminology used in DEVRAW Docs & throughout Mod Tools.

Engine Functionsâ€‹

Engine functions are functions that are pre-defined, meaning that they're built into BO3. They can be called anywhere, regardless of variable scope.

For example: GetEnt(), GetEntArray(), SetHintString(), GetVelocity(), etc.

You can find a collection of all engine functions on GSCode's API page

Syntaxâ€‹

The rules of the language. In English, not capitalising proper nouns, or missing out full stops to mark the end of the sentence is breaking the syntax of the English language.

Same goes for programming. In GSC, for example, missing out the semicolon ; to mark the end of a line breaks the syntax. While someone fluent in the language may still be able to figure out what you mean, computers aren't very clever, and rely on this syntax to understand what you've written correctly.

Syntax errors for GSC will show in the launcher log.

Context Menuâ€‹

A "context menu" refers to the menu that comes up when you right click something.

...you'll be surprised how many people don't know what a context menu is.

3arcâ€‹

Shorthand for Treyarch, the main developer studio that worked on Black Ops III.

note

All Treyarch games have a code name of TX, where X is the number of games they've made. E.g. Black Ops III was referred to as T7 during development.

Lots of files and assets made for BO3 start with t7_. This is most likely done to to differentiate which assets were made for which game in CoD's studios, as the game names are often not finalised until the late stages of development.

What's funny is you can actually find a lot of T7 assets in T9 (Black Ops Cold War) with Greyhound, as development time was short for that game, leading to some assets being reused. And we know they're BO3 assets because of the t7_ prefix.

This is a standard CoD naming convention. The other studios also do this (CoD Ghosts' is "IW6" and Advanced Warfare is "S1").

CSV Filesâ€‹

Comma Separated Value (CSV) files are commonly used within the Mod Tools for storing data in a table format.

XPak Filesâ€‹

XPaks are a type of file that stores the majority of assets for a map. This includes (but is not limited to): textures, models, animations, etc.

For usermaps, they're stored at <BO3 Root>\usermaps\<MapName>\zone\*.xpak. The base game stores them in <BO3 Root>\zone\*.xpak.

Cleaning XPaksâ€‹

When the linker builds a map that already has an existing XPak, it will append any new assets to the end of the XPak.

This turns into an issue when you've previously built an asset into your map, and you're no longer using it anymore. Rebuilding your map doesn't remove any of the old assets that aren't used anymore.

Let's say you made a new map, added two static custom models and built the map. The linker would add all the data (textures, XModel meshes, etc.) to the map's XPak.

Then you go back into Radiant, delete one of the models and rebuild the map. Even though the model doesn't show in-game, the XPak will still have that asset in.

One asset isn't the end of the world, but this can add up and inflate your map size.

Make sure you're cleaning XPaks in the launcher before publishing your map.
