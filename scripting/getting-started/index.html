Getting Started - GSC

This is all you need to know to get started in Black Ops III GSC.

You'll be downloading Visual Studio Code, setting it up for use in BO3's Mod Tools, and understanding essential Radiant basics to help get you started.

Beginner

What is GSC?​

GSC (or Game Script Code) is the scripting language used by Call of Duty. GSC is Call of Duty's in-house scripting language - it's not used by the general public; it's specifically made for CoD scripting, unlike other programming languages (C/C++, Python, Java, etc.), which could be used for virtually anything. Languages like these are usually referred to as scripting languages than fully-fledged programming languages. It's built around Call of Duty's engine, each game they usually iterate on the language and change things. DEVRAW's documentation is based around T7's GSC.

For those familiar with the C family of languages (C/C++, C#), they'll notice that the syntax is similar to GSC. GSC is inspired by C++ and C#, implementing elements from both languages. But they're not the same.

Fundamentals​

If you already know a programming language, then you'll find it much easier to pick up GSC.

If you have no programming experience, there are a few things you need to learn before you can understand GSC, like:

  • What variables are
  • What different data types we store in these variables (strings, integers, arrays, etc. )
  • Functions and how they work (calling functions, function pointers)
  • Variable scopes (local vs global variables)
  • Passing arguments by reference vs passing by value
  • Object-oriented programming (mainly just storing variables in objects)

The list goes on. As long as you understand what all of these are, then you should be good to move on. If you don't know one or more of these, I recommend you find videos explaining these on YouTube.

However, I've made GSC-specific explanations on the programming fundamentals page as well.

Interested in programming?

This is completely optional, but if you're interested in learning programming as a skill in general and not just for your map, I recommend learning the above concepts in Python first. Try doing a few exercises or make some simple programs. Some might seem really easy once you understand more about programming, but you should do them anyway because your aim is to get confident and familiar with the concepts mentioned above, and that comes with practice.

Then after a while, maybe a week or two, once you're a bit more confident with programming, you can bring what you've learned over to GSC. Python is similar to english, so I've found it's easy to understand for people new to programming (it's what I started off with). Python's easy-to-understand syntax will allow you to focus more on understanding the basics of programming that you can apply to other languages. Things won't be exactly the same as other languages, but you'll definitely recognise places in the programming world where people have been influenced by Python's way of doing things. However, the syntax of Python is not the same as GSC, so bear that in mind.

Learning your first programming language is hard. Learning your second is much easier, and learning your third takes no more than a weekend. Python will make it much easier to learn that first language. You'd essentially learn Python first to make the learning curve feel shallower.

Programming Environments for GSC​

The first thing we'll need is somewhere to write code. There are programs to help you do this called "Integrated Development Environments" (IDEs).

For GSC, because it's not a very popular or supported language, we only have a few options:

Visual Studio Code or Sublime Text

Visual Studio Code is what I recommend. If you already have it, skip ahead to here.

The best IDE for GSC is Visual Studio Code (or VS Code). It can be used for many languages and is praised for how lightweight it is. It's literally what I'm using right now to construct this docs page.

But the reason it's the best option for GSC is because Sublime, the other alternative (other than notepad, but ISTG IF YOU USE NOTEPAD I WILL HUNT YOU DOWN) isn't the best because its only feature is syntax highlighting given to us by Treyarch.

Sublime is limited in general in various other aspects when compared to VS Code, but it's still a powerful editor.

TL;DR: VS Code > Sublime.

Installing & setting up VS Code​

Again, if you already have VS Code, you can skip ahead to here.

You can get VS Code on their website code.visualstudio.com and install the latest version.

Run the installer (.exe) you just downloaded and follow the setup. When you get to the "Additional Tasks" page, make sure to select the following:

This will make it easier for you to open files and folders later. You can continue with the rest of the setup.

Once you select "Finish" on the setup, VS Code is now installed.

Setting VS Code as the default app for Mod Tools files​

You should set the default program for GSC files to VS Code so that whenever you open your scripts, they open in VS Code automatically.

It'll take 2 minutes now, and save you loads of time and hassle later.

There are two ways to do this:

First way (going to each file type)​

  • Go to any GSC file, you'll be able to find one in your map folder (right click any zombies map in the launcher & click "Open map folder") in <BO3 ROOT>\usermaps\<MAP NAME>\scripts\zm\*. There should be two files in there with the same name as your map name, one a .gsc file, one a .csc.

  • Right click on the file, expand the "open with" context menu & click "Choose another app".

    Don't get confused

    If you see an "Open with Code" option in the context menu, that's the wrong one!

    This won't set VS Code as the default application, rather it'll open in VS Code just once. Make sure you follow the screenshot above.

  • Look for "Visual Studio Code" and then click it and click "Always" (If you're on Windows 10 you'll see a checkbox, just make sure it's checked)

  • Now you need to repeat this for the other file types that are native to Mod Tools. You can find a list of file paths for each file type below:

File TypeExample file path (relative to BO3's root)
.gscshare\raw\scripts\zm\_art.gsc
.cscshare\raw\scripts\zm\_ambient.csc
.gshshare\raw\scripts\zm\_bonuscard.gsh
.zoneusermaps\zm_giant\zone_source\zm_giant.zone
.zpkgshare\zone_source\base_patch.zpkg

Second way (through Windows Settings)​

  • Press Windows Key + I to open Windows Settings.
  • Search for the "default apps" page.
  • If you're on Windows 11, you'll be able to just search for a file type (it's the top search box), however Windows 10 doesn't have a search bar to search file types in this settings page. They're ordered alphabetically, so just scroll to each file type instead.
  • Find the file type .gsc and set it to "Visual Studio Code". Do the same thing for .csc, .gsh, .zone, .szc & .zpkg.
tip

Sometimes the file extension won't show up in settings unless you've tried opening a file before with that extension. If you can't find the file extensions, you're gonna have to use the first way

Adding GSC support - Installing Ardivee's Zoroth​

Outside of Mod Tools, GSC is CoD's proprietary (in-house) scripting language, so there isn't native support for it in any software. So it's not supported by VS Code off the bat. You'll need a GSC extension. Most of the extensions, however, don't provide a lot of features.

You can browse the extensions tab in VS Code for a GSC extension, but the best extension I've found is Ardivee's Zoroth, which isn't on the extensions marketplace, so you'll have to download it here & install it manually (see Installing Zoroth below).

One of the features of this extension is it comes with a language parsing server. Basically, it reads the scripts in your share\raw\scripts folder, and knows the parameters and functions from those scripts. So you can hover over functions and see what arguments they take, CTRL + click functions and jump to them, and more.

It also tells you all mandatory and optional arguments, even for engine functions.

You might not understand the benefits of these features now, but I promise you it will be extremely useful later on when you start writing your own scripts. As of writing this, no other GSC extension has a language server, which is why I've recommended Zoroth, made by Ardivee.

Giving credit where it's due

ALL credit for this extension goes to Ardivee. He made it, so a huge thank you to him. DEVRAW takes no credit for his work. Ardivee's Wiki is also linked on the bottom of this site, so make sure you check that out also.

Installing Zoroth​

  1. Extract Zoroth's .zip file from the link above into any folder.

    Zip files

    If you're using WinRAR (or you don't have a .zip extractor), get 7-Zip instead. 7-Zip is better because it has better context menu options than WinRAR. Also that "license expired" pop up on WinRAR is annoying, so save yourself the headache and just use 7-Zip. Windows does have a built-in zip extractor, but you can easily get confused between when you're in a folder vs. when you're in a .zip file, so I'd just stick to 7-zip as a general rule of thumb.

  2. Open VS Code. On the left, you'll see the extensions tab, click the icon with 4 cubes, then click the ellipsis in the top right of the extensions sidebar. (See picture below.)

    At the bottom of the context menu, click "Install from VSIX...", and then go to the place you extracted that .zip file. There should be a .vsix file, so select that.

    You should see a message in the bottom right of VS Code telling you it completed the installation.

You should now have Zoroth installed.

Workspaces in VS Code​

VS Code has a feature where if you have tabs open and then close the program, the tabs will still be there when you reopen the program (a bit like the "continue where you left off" feature in browsers). However, if you're not in a workspace, you'll lose these tabs if you open VS Code by opening a file.

Now that you have VS Code installed, you're gonna be opening loads of different files, and this'll happen all the time. It can get quite frustrating.

This is one of the reasons we use workspaces. VS Code has their own documentation on workspaces, but to a beginner, their explanation on what it is and why it's useful can be confusing. If you're comfortable with using VS Code and you don't fully understand workspaces, then I recommend you check out their official docs page on workspaces.

It's good practice to open the BO3 root folder in VS Code when we're opening files and/or writing scripts. This helps us not only save our tabs, but keep organised if we use VS Code for other things outside of the tools. To open a folder in VS Code, you can either go to File -> Open Folder... or do CTRL + K, then CTRL + O straight after.

I recommend if you only work on one map at a time, then you might as well just open the BO3 root folder, which would set that as your current workspace. However, another option is to open the usermaps folder for the map you're working on, as that'll allow you to have all your map-specific scripts separated into their own workspaces. So you essentially have different workspaces (and hence a different group of tabs) for each of your maps, which helps if you frequently switch projects.

EXAMPLE

For example, if I was working on a map called zm_subway, I'd do CTRL + K CTRL + O in VS Code and select the <BO3 ROOT>\usermaps\zm_subway folder to set that as my workspace. Then I can open up my zm_subway.gsc and csc scripts either through VS Code's built-in explorer on the left or just by opening the files through file explorer. As long as VS Code is open, it'll just open them as a new tab in the editor.

You can also have multiple workspaces open at once by opening a new VS Code window. Either select File -> New Window, or do CTRL + SHIFT + N in the editor.

Whether you use your BO3's root folder, or individual usermap folders as workspaces, this will save you time when you re-open VS Code inside a new workspace, or through opening a file. Because now you'll find your tabs will still be there when you open the same folder again, so you don't have to re-open all your files.
