GSC Constructs

A detail on GSC's sequence, selection and iteration statements, and how they work.

Beginner

Control structures just define when code is skipped, or ran more than once by the computer. Usually when a certain condition is met.

Sequential statements​

This just means that code is ran from top to bottom. Selection and iteration statements allow us to break this rule (see below).

Selection statements​

Sometimes we only want to run code if certain conditions are met.

For example, imagine a script that lets the player buy a door when they have enough points. You want to run the code that opens the door up IF the player has enough points.

A selection statement checks if something is true or false. You can control which code runs depending on whether a condition is satisfied.

if-else statements​

Selection Statements
if( player JumpButtonPressed() )
{
    // Run all code in here if above statement is true
    IPrintLnBold( player.name + " has just pressed the JUMP button!" );
    
    // do more stuff...
}
else if( player SprintButtonPressed() )
{
    // If previous statement was false, and if this statement is true...
    IPrintLnBold( player.name + " has just pressed the SPRINT button!" );
}
else if( player AttackButtonPressed() )
{
    // Check this statement if the previous was false
    // (they're always checked in order)
    IPrintLnBold( player.name + " has just pressed their ATTACK button!" );
}
else
{
    // If none of the statements above were true, then run this code
    IPrintLnBold( player.name + " did not sprint, jump or attack :/" );
}
You don't always need to use else if or else...
if( !isdefined( player ) )
{
    return;
}

// Just one check!

This is also perfectly valid code! No need to always have an else if or else statement.

One-line if statements

For if statements with only one line of code in its block, you don't need to wrap the one line curly braces.

if( !isdefined( player ) )
    return;

You can also do this with for / foreach / while statements

foreach( player in GetPlayers() )
    player zm_score::add_to_player_score( 420 );

If you do not use any curly braces {}, then the GSC compiler will assume that ONLY the next line belongs to the if statement.

The "next line" is everything up to the first semicolon (;) after the if statement, as a semicolon is seen as the end of the line by the GSC compiler. Indentation and lines are ignored by the GSC compiler, (but still preserved for error messages).

So, does that mean you can also do this?

if( !isdefined( player ) ) return;

The answer is... yes! You can do that. But it's not recommended, as it can make your code harder to read. When following the logic of a program, people expect to see a new line, with some sort of indentation to signify that the code is part of the if statement. So, it's good practice to use the first example I showed you.

Don't make this mistake!

Consider this code that doesn't break out of the loop until the player has enough points:

for( ;; ) // Infinite loop
{
    door_trigger waittill( "trigger", player );

    // script_int is set in Radiant & is used as the price of the door here
    if( player.score != door_trigger.script_int )
        continue; // Go back to the top of the for loop (cause they're a brokie)
    
    IPrintLnBold( "Door Purchased!" );
    break;
}

// Rotate door model & take points from player, blah blah blah...

A common error is to write a one-line if statement, and then want to add something to it and forget to put the brackets back.

Let's add an IPrintLnBold so I can demonstrate the error we might make:

for( ;; )
{
    door_trigger waittill( "trigger", player );

    if( player.score != door_trigger.script_int )
        IPrintLnBold( "Not enough points!" ); // <--- if statement ends here
        continue; // This is considered outside of the if statement!
    
    IPrintLnBold( "Door Purchased!" );
    break;
}

The GSC compiler sees the above if statement like this:

    if( player.score != door_trigger.script_int )
    {
        IPrintLnBold( "Not enough points!" );
    }

    continue;

If you can't see the problem here, let me outline it for you:

The continue statement is outside the if statement, so the function will loop back to the top, regardless of whether that if statement is true.

Therefore, the script will never reach the "Door Purchased!" & break statements. This will break functionality of the script, as every time you activate the trigger in-game, it will just run that if statement, then branch back to the top and start waiting for the next trigger.

You will get no errors pointing to the issue. Be weary of this.

Switch Cases​

if-else statements are one way to run different code based on if certain conditions are met.

Another way is by using a switch-case statement:

Switch-case Statement
// <weapon_obj>.name gets the console name, if you want the in-game name, do <weapon_obj>.displayname
wpn_name = player GetCurrentWeapon().name;
player.has_starting_weapon = undefined;

switch( wpn_name )
{
    case "pistol_standard": // Equivalent to: if( wpn_name == "pistol_standard" )
    {
        IPrintLn( "You have the starting pistol!" );
        player.has_starting_weapon = true;

        break;
    }
    
    case "ar_peacekeeper": // Equivalent to: if( wpn_name == "ar_peacekeeper" )
    {
        IPrintLn( "You have the Peacekeeper!" );
        player.has_starting_weapon = false;

        break;
    }

    case "launcher_multi": // Equivalent to: if( wpn_name == "launcher_multi" )
    {
        IPrintLn( "You have that really cool rocket launcher gun, noice." );
        player.has_starting_weapon = false;

        break;
    }

    default: // Equivalent to "else" - will run if none of the above cases are met
    {
        IPrintLn( "I wonder what weapon you're holding..." );
        player.has_starting_weapon = false;

        break;
    }
}

The default case is optional. If you don't include it, then the script will just move on to the next line of code if none of the cases are met. The curly braces surrounding each case's code block are also optional, but it's good practice to include them.

Switch case vs. if statements: why use one over the other?

You can mimic the above switch-case logic with if statements instead. But then the question arises: Why do switch-cases exist?

The switch-case statement is a more readable way to write multiple if-else statements. It's also more efficient than using multiple if-else statements, as the program will only check the value of wpn_name once, then run the code in the corresponding case.

They were implemented in C as a more optimised alternative when comparing one variable lots of times. A general rule you can follow is if you're comparing the same item more than 5 or so times, then it'll be better to use a switch-case over an if statement.

Here is equivalent if statement logic of the above switch-case:

Switch-case Equivalent
wpn_name = player GetCurrentWeapon().name;
player.has_starting_weapon = undefined;


if( wpn_name == "pistol_standard" ) // Equivalent to: case "pistol_standard":
{
    IPrintLn( "You have the starting pistol!" );
    player.has_starting_weapon = true;
}
else if( wpn_name == "ar_peacekeeper" ) // Equivalent to: case "ar_peacekeeper":
{
    IPrintLn( "You have the Peacekeeper!" );
    player.has_starting_weapon = false;
}
else if( wpn_name == "launcher_multi" ) // Equivalent to: case "launcher_multi":
{
    IPrintLn( "You have that really cool rocket launcher gun, noice." );
    player.has_starting_weapon = false;
}
else // Equivalent to: default:
{
    IPrintLn( "I wonder what weapon you're holding..." );
    player.has_starting_weapon = false;
}

As you've probably noticed, the switch case looks more readable than the if statements.

REMINDER

Don't forget to add the break; statement at the end of EACH CASE, including the default case at the end if you put it.

If you forget, your map will crash when the script reaches the switch case.

Iteration​

To "iterate" just means to repeat something. Also known as "loops".

With iteration in programming, you can run a block of code and then repeat that same codeblock multiple times.

There are two types of iteration:

Count-controlled loops​

These loops run a certain amount of times before continuing on with the rest of the script.

To use a count-controlled loop, you can either use a for loop or a foreach loop in GSC.

Example of a for loop
for( idx = 0; idx < 4; idx++ )
{
    IPrintLn( idx );
}

/*
Output:
0
1
2
3
*/

Let's break down that first line:

for( idx = 0; idx < 4; idx++ )

idx = 0; - This is the init expression - it's executed first before the rest of the for loop. It's defining the variable idx, setting it to 0. The idx variable is your index, and we use this to keep track of how many times we've looped.

idx < 4; - This is the condition - it's checked before each iteration of the loop. If it's true, the loop will run. If it's false, the loop will stop.

idx++ - This is the increment expression - it's executed after each iteration of the loop. It's used to increment the index variable idx by 1.

The for loop will run the code block inside the curly braces {} until the condition idx < 4 is false.

Using for loops with arrays​

for loops can also be used to iterate over arrays.

scripts/zm/_zm_utility.gsc --> get_current_zone()
for (i = 0; i < zone.volumes.size; i++)
{
	if ( i == vol )
		continue; 
	if ( self IsTouching(zone.volumes[i]) )
	{
		self.cached_zone = zone; 
		self.cached_zone_volume = i; 
		if ( IS_TRUE( return_zone ) )
		{
			return zone;
		}
		return zone_name;
	}
}
