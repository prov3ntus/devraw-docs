GSC Fundamentals

This page outlines essential information on programming that every scripter needs to know before writing in GSC.

Beginner

Everything listed here is essential to know before you start writing scripts in GSC. If you're new to programming, then this is the place to start.

Data Types​

There are lots of different data types. Some are exclusive to some languages, others are shared with all languages.

In GSC, the main data types are:

Data TypeDefinitionExamples
StringA sequence of characters / symbols / digits; a piece of text, surrounded by quotes ("), unless it's in a variable. The engine function IsStr( <arg> ) will return true if <arg> is a string."This is a string!", "10", "trigger_activated"
IntegerA whole number. Can be positive or negative. No fractional values.-128, 69420, 0, 1, 115
Floating pointA decimal / fractional number. Can be positive or negative. Whole numbers are also allowed.15.67525, 3.1415926, 4.0, .05, 12., .0
BooleanTrue or false values. Boolean values can be treated as 1 or 0 thanks to GSC being a dynamically-typed language. It's recommended not to use 1 instead of true (and vice versa) for clarity and readability (you'll also get mocked by any decent programmer that sees you use 1 instead of true for boolean values).true, false
ArrayA list of items in an index (a specific order), structured as key-value pairs. Default index keys is 0 to n - 1, where n is the length of the array. Keys can be any data type, and must be unique. In GSC, they can hold multiple data types at once. Similar to Python's dictionaries / lists.my_array = []
VectorA group of three float values. Commonly used to store transforms, i.e. an object's position (origin) or its angles. Transforms are in the format ( x, y, z )player.origin, player.angles, `( 126, )

Variables​

Variables (in programming) are like little containers that can store information for you. They enable developers to store something away in RAM, label it with a name, and then access it later. Just like storing something in a labelled box.

If I wanted to store a value, I'd stick it in a box and slap a label on it. Then if I wanted to retrieve a value from a box, I'd find the box with the correct label on it and take the value back out again.

The place we store all our boxes is the computer's memory (the RAM), and the labels we give the boxes are our variable names. Finding the box and reading the value is the equivalent of accessing and calling the variable.

info

Variable names can't have spaces or special characters in, and can't start with a number.

Variables can change what they're storing while the code is running (during runtime). In some programming languages, when you redefine a variable, the type of data that's in that variable can't change. So you can't create a variable that stores an int value of 4, then try to redefine the variable and put a float (decimal) value of 6.5 in there.

These types of programming languages are referred to as static languages. For these languages, the data types of variables are defined at compile time and can't be changed during runtime. You can still change what data is in the variable, just not the type.

In dynamically-typed languages, however, you can redefine variables with whatever you want and don't have to worry about data types. GSC is a dynamically-typed language.

Does capitalisation matter?

When typing function or variable names, does it matter if I put player.name or player.Name? Or even PLAYER.name? The answer is NO, in GSC they're the same variable. But this is pretty much unique to GSC.

GSC is one of the rare languages that is case insensitive, meaning that function names, variable names, etc. can be either capitalised or not, it doesn't matter. Fully capitalised variables are seen as constants though (see below).

Caution when using strings!

Strings are always case-sensitive, no matter what language you're writing. For example, the string "Hello World" is NOT the same as "hello world".

Constants​

Usually when you see fully CAPITALISED variables in a script, it's a macro / constant. Constants are variables that don't change what they're storing. In general for programming, it's widely accepted that constants are fully capitalised to show that they're a constant. Otherwise, when we call it, it's often hard to tell whether it's a variable or a constant.

Defining a constant:

share\raw\scripts\zm\zm_weapons.gsc --> get_ammo_cost_for_weapon()
const N_WALLBUY_UPGRADE_COST = 4000;

You can define and call a constant without naming it in full caps, but it's bad practice to do so, as explained above.

The reason we use constants is because they can be more efficient than variables; the program knows exactly what that piece of information is because it can't be redefined, so it doesn't have to wait while it's fetched from memory. Therefore constants can be slightly faster to use than variables. They are also a way to communicate to other developers (or even yourself in the future) that this memory should not be redefined. If another developer (or your future self) tries to redefine the constant, an error will be thrown.

However, it's completely optional to use constants. Don't think that you have to make every variable that isn't redefined a constant. The only reason I've included them in this documentation is so that if someone wants to understand them, they can do so here.

In the stock scripts we were given in Mod Tools, there are about 120 constants defined, whereas there are over 20,000 variables defined. Therefore, variables are used over 160 times more than constants. In other words, you don't need to use constants very often if at all.

While constants still suffer from their function-bound variable scope, a macro is accessible script-wide. Macros are used in specific scenarios and can often be more useful than constants. Read the macros section here for more info on that.

Operators​

Operators are just symbols that perform math or comparison operations on values.

Arithmetic operators let you do math calculations.

Comparison operators let you compare two values.

Arithmetic Operators​

You can use arithmetic operators to perform math operations on numbers (either int or float data types).

Arithmetic Operators
total = 1000 + 250; // Add
time = 5 - 2; // Subtract
distance = 420 * -1; // Multiply
speed = distance / time; // Divide

Because GSC is a dynamically-typed language, the data types are handled for you. You can mix integers and floats together in operations and not worry about it.

Dividing any two integers will always return a float.

Example of integer division: Calculating player's K/D
kd_ratio = player.kills / player.deaths;
if( IsFloat( kd_ratio ) )
    IPrintLn( "Float!" );
else if( IsInt( kd_ratio ) )
    IPrintLn( "Integer!" );

/*
Output: Float!

Using + with strings

Yo can also use the + operator to add strings together. This is called concatenation.

trigger waittill( "trigger", player );
IPrintLnBold( player.name + " just activated the trigger, with !" );

This also works with number types, and it's how we convert numbers to strings in GSC.

// Convert the int 12 to a string
str_player_points = "" + player.points; // Doing IsStr( player.points ) will return true
Operator order

The GSC VM, like all other programming languages, follows the order of operations BIDMAS / BODMAS. This means that it will evaluate operations inside brackets () first, then it will evaluate division and multiplication, and then addition and subtraction.

Selection statements​

A selection statement checks if something is true or false. You can control which code runs depending on if a condition is satisfied/true.

if-else statements​

Selection Statements
if( player JumpButtonPressed() )
{
    // Run all code in here if above statement is true
    IPrintLnBold( player.name + " has just pressed the JUMP button!" );
    
    // do more stuff...
}
else if( player SprintButtonPressed() )
{
    // If previous statement was false, and if this statement is true...
    IPrintLnBold( player.name + " has just pressed the SPRINT button!" );
}
else if( player AttackButtonPressed() )
{
    // Check this statement if the previous was false
    // (they're always checked in order)
    IPrintLnBold( player.name + " has just pressed their ATTACK button!" );
}
else
{
    // If none of the statements above were true, then run this code
    IPrintLnBold( player.name + " did not sprint, jump or attack :/" );
}
You don't always need to use else if or else...
if( !isdefined( player ) )
{
    return;
}

// Just one check!

This is also perfectly valid code! No need to always have an else if or else statement.

One-line if statements

For if statements with only one line of code in its block, you don't need to wrap the one line curly braces.

if( !isdefined( player ) )
    return;

You can also do this with for / foreach / while statements

foreach( player in GetPlayers() )
    player zm_score::add_to_player_score( 420 );

If you do not use any curly braces {}, then the GSC compiler will assume that ONLY the next line belongs to the if statement.

The "next line" is everything up to the first semicolon (;) after the if statement, as a semicolon is seen as the end of the line by the GSC compiler. Indentation and lines are ignored by the GSC compiler, (but still preserved for error messages).

So, does that mean you can also do this?

if( !isdefined( player ) ) return;

The answer is... yes! You can do that. But it's not recommended, as it can make your code harder to read. When following the logic of a program, people expect to see a new line, with some sort of indentation to signify that the code is part of the if statement. So, it's good practice to use the first example I showed you.

Don't make this mistake!

Consider this code that doesn't break out of the loop until the player has enough points:

for( ;; ) // Infinite loop
{
    door_trigger waittill( "trigger", player );

    // script_int is set in Radiant & is used as the price of the door here
    if( player.score != door_trigger.script_int )
        continue; // Go back to the top of the for loop (cause they're a brokie)
    
    IPrintLnBold( "Door Purchased!" );
    break;
}

// Rotate door model & take points from player, blah blah blah...

A common error is to write a one-line if statement, and then want to add something to it and forget to put the brackets back.

Let's add an IPrintLnBold so I can demonstrate the error we might make:

for( ;; )
{
    door_trigger waittill( "trigger", player );

    if( player.score != door_trigger.script_int )
        IPrintLnBold( "Not enough points!" ); // <--- if statement ends here
        continue; // This is considered outside of the if statement!
    
    IPrintLnBold( "Door Purchased!" );
    break;
}

The GSC compiler sees the above if statement like this:

    if( player.score != door_trigger.script_int )
    {
        IPrintLnBold( "Not enough points!" );
    }

    continue;

If you can't see the problem here, let me outline it for you:

The continue statement is outside the if statement, so the function will loop back to the top, regardless of whether that if statement is true.

Therefore, the script will never reach the "Door Purchased!" & break statements. This will break functionality of the script, as every time you activate the trigger in-game, it will just run that if statement, then branch back to the top and start waiting for the next trigger.

You will get no errors pointing to the issue. Be weary of this.

Switch Cases​

if-else statements are one way to run different code based on if certain conditions are met.

Another way is by using a switch-case statement:

Switch-case Statement
// <weapon_obj>.name gets the console name, if you want the in-game name, do <weapon_obj>.displayname
wpn_name = player GetCurrentWeapon().name;
player.has_starting_weapon = undefined;

switch( wpn_name )
{
    case "pistol_standard": // Equivalent to: if( wpn_name == "pistol_standard" )
    {
        IPrintLn( "You have the starting pistol!" );
        player.has_starting_weapon = true;

        break;
    }
    
    case "ar_peacekeeper": // Equivalent to: if( wpn_name == "ar_peacekeeper" )
    {
        IPrintLn( "You have the Peacekeeper!" );
        player.has_starting_weapon = false;

        break;
    }

    case "launcher_multi": // Equivalent to: if( wpn_name == "launcher_multi" )
    {
        IPrintLn( "You have that really cool rocket launcher gun, noice." );
        player.has_starting_weapon = false;

        break;
    }

    default: // Equivalent to "else" - will run if none of the above cases are met
    {
        IPrintLn( "I wonder what weapon you're holding..." );
        player.has_starting_weapon = false;

        break;
    }
}

The default case is optional. If you don't include it, then the script will just move on to the next line of code if none of the cases are met. The curly braces surrounding each case's code block are also optional, but it's good practice to include them.

Switch case vs. if statements: why use one over the other?

You can mimic the above switch-case logic with if statements instead. But then the question arises: Why do switch-cases exist?

The switch-case statement is a more readable way to write multiple if-else statements. It's also more efficient than using multiple if-else statements, as the program will only check the value of wpn_name once, then run the code in the corresponding case.

They were implemented in C as a more optimised alternative when comparing one variable lots of times. A general rule you can follow is if you're comparing the same item more than 5 or so times, then it'll be better to use a switch-case over an if statement.

Here is equivalent if statement logic of the above switch-case:

Switch-case Equivalent
wpn_name = player GetCurrentWeapon().name;
player.has_starting_weapon = undefined;


if( wpn_name == "pistol_standard" ) // Equivalent to: case "pistol_standard":
{
    IPrintLn( "You have the starting pistol!" );
    player.has_starting_weapon = true;
}
else if( wpn_name == "ar_peacekeeper" ) // Equivalent to: case "ar_peacekeeper":
{
    IPrintLn( "You have the Peacekeeper!" );
    player.has_starting_weapon = false;
}
else if( wpn_name == "launcher_multi" ) // Equivalent to: case "launcher_multi":
{
    IPrintLn( "You have that really cool rocket launcher gun, noice." );
    player.has_starting_weapon = false;
}
else // Equivalent to: default:
{
    IPrintLn( "I wonder what weapon you're holding..." );
    player.has_starting_weapon = false;
}

As you've probably noticed, the switch case looks more readable than the if statements.

REMINDER

Don't forget to add the break; statement at the end of EACH CASE, including the default case at the end if you put it.

If you forget, your map will crash when the script reaches the switch case.

Iteration​

To "iterate" just means to repeat something. Also known as "loops".

With iteration in programming, you can run a block of code and then repeat that same codeblock multiple times.

There are two types of iteration:

Count-controlled loops​

These loops run a certain amount of times before continuing on with the rest of the script.

To use a count-controlled loop, you can either use a for loop or a foreach loop in GSC.

Example of a for loop
for( idx = 0; idx < 4; idx++ )
{
    IPrintLn( idx );
}

/*
Output:
0
1
2
3
*/

Let's break down that first line:

for( idx = 0; idx < 4; idx++ )

idx = 0; - This is the init expression - it's executed first before the rest of the for loop. It's defining the variable idx, setting it to 0. The idx variable is your index, and we use this to keep track of how many times we've looped.

idx < 4; - This is the condition - it's checked before each iteration of the loop. If it's true, the loop will run. If it's false, the loop will stop.

idx++ - This is the increment expression - it's executed after each iteration of the loop. It's used to increment the index variable idx by 1.

The for loop will run the code block inside the curly braces {} until the condition idx < 4 is false.

Using for loops with arrays​

for loops can also be used to iterate over arrays.

scripts/zm/_zm_utility.gsc --> get_current_zone()

Functions​

A function just houses a bunch of code together. Developers will separate logic for specific tasks into separate functions.

Defining a function in GSC:​

Simple function declaration
function my_new_function()
{
    IPrintLnBold( "Hello world!" );
}

Functions helps separate code out, making it easier to follow and understand. Not only that, it allows us to re-run the code in that function, instead of re-writing it.

For example, if you have 12 different buyable doors in your map, you can create one function that will get called on every single door. Then any properties that change for each debris, i.e. the purchase price, or the specific door model that needs to be swung open, you can pull from KVPs in Radiant. KVPs get passed into scripts as variables, but they're stored in the entity's object.

Calling a function:​

Example function call scenario
function main()
{
    // Get entity with targetname "debris_trigger" from the level
    debris_trig = GetEnt( "debris_trigger", "targetname" );
    debris_trig trigger_logic(); // Call function on trigger
}

// self == a debris trigger
function trigger_logic()
{
    self SetHintString( "Activate Me!" );
    // ...
}

As you can see, we've ran the trigger_logic() function on the debris_trig object that we got from the level using the GetEnt() engine function. Now we can access the debris_trig entity as self in the trigger_logic() function. Objects like entities returned by GetEnt() are passed by reference. So, self in the trigger_logic() function will point to the original object.

The SetHintString() and the GetEnt() function is an engine function. You can find most of the engine functions documented on gscode.net .

Code readability​

I thought I'd stick this section at the end, not because it's unimportant, but because it's more of a "best practice" than a fundamental.

Code readability is important because it makes it easier for you to debug your code, and for others to understand it.

But forget other people for a second. If you leave a script for a few months and then come back to it, you're going to have a really hard time if its formatted poorly. Sometimes, you look back at code and you don't remember what you were thinking when you were writing it, if you even remember writing it at all. So, it's important to make your code readable to make it easier for your future self to understand.

Don't be a lazy programmer, and please, for the love of god, format your code properly.

Naming conventions​

There are three main types of naming conventions in programming:

  1. Snake Case (AKA Underscore Notation):

    • In Snake Case, multiple words are separated by underscores (_), and each word is written in lowercase.
    • Examples: my_variable, get_player_speed.

  2. Camel Case:

    • In Camel Case, the first word is written in lowercase, and subsequent words start with uppercase letters.
    • Examples: myVariable, getPlayerSpeed.

  3. Pascal Case (also known as Initial Caps):

    • In Pascal Case, each word starts with an uppercase letter.
    • Examples: MyVariable, GetPlayerSpeed.

In GSC, the standard for variable and function names is snake_case.

However, there are some exceptions to this rule: rarely, variables created by 3arc devs in the stock scripts are in camelCase (and some don't have even any spacing or capitalisation at all 💀). Not that capitalisation matters, because GSC is case insensitive anyway.

Furthermore, the standard for Engine functions is different. Engine functions use PascalCase.

Example of how different naming conventions are used in GSC
if( !IsAI( self.attacker ) ) // IsAI() is an engine function, so uses PascalCase
{
    zombie_team = undefined; // snake_case for zombie_team
    zm_utility::recalc_zombie_array(); // snake_case for recalc_zombie_array (and the zm_utility namespace I guess)
}

Spacing​

One thing I hate more than anything is reading code with no spaces in. It's honestly comparable to reading a book with no spaces between the words. If I'm learning a new Python or C++ concept, I'll often look at Stack Overflow or some other forum to learn about it.

Sometimes, people will post code snippets like this:

Poorly formatted C++ code
if(x<0.3&&y>2||!std::rand(x-y)&&!condition){
    std::cout<<++x?0:1<<std::endln;
}

Now, compare that with this:

Well formatted C++ code
if(x < 0.3 && y > 2 || !std::rand(x - y) && !condition)
{
    std::cout << ++x ? 0 : 1 << std::endln;
}

You can tell me which one you'd rather read.

There is a balance, though. You don't want to go overboard with spaces. For example, you don't need to put a space between a function name and its calling parentheses.

There are variations of this, with some preferring not to put spaces between brackets and their contents. You've probably noticed from all the GSC code snippets I wrote that I personally do, but it's up to you.
